# 開発ガイドライン

このドキュメントは、プロジェクトの現状と運用方針を示す北極星ドキュメントとして扱う。

## プロジェクト概要

- このプロジェクトは Next.js 14 と React 18 を使用する Web アプリケーションである。
- 現在のアプリケーションは、川辺コンピューターとかわべ家庭教師学院の2つの専門サービスへ利用者を案内する「KAWABIT（カワビット）」ブランドのハブサイトである。
- トップページでは、パソコン修理・データ復旧の相談先として `https://kawabecom.jp/`、プロ家庭教師・学習相談の相談先として `https://kawabegakuin.jp/` へ誘導する。
- サイトデザインは `design.md` を正本とし、白〜生成りの余白、黒の極太ライン、2〜4色程度のソリッドなフラットカラー、ミニマルなコミック風PNGイラストを基調とする。
- トップページのFVは、左にメインコピー「学びとITの相談窓口。」と2つの導線、右に家庭教師・PC修理を想起させる小さなポイントイラスト群を配置し、写真ではなく余白とイラストの間で KAWABIT の入口感を表現する。
- トップページの続きは、すべてをイラストで説明せず、見出し・本文・CTAを主役にしながら、各セクションへ極太ラインのPNGイラストをポイントとして配置する。
- FVはヘッダーを除いた初期表示領域を基本にしつつ、モバイルでは縦積みで読みやすさを優先する。
- 表示言語は日本語を基本とし、問い合わせ時間は `10:00-19:00 / 日曜休` として表示する。
- アプリケーションコードは主に `app/` に配置する。
- 静的アセットは `public/` に配置する。
- 画像や素材の元データは、必要に応じて `asset-sources/` に配置する。

## ビジュアルアセット運用ルール

- サイト内のイラストは PNG 形式のラスター画像として生成・配置する。
- SVG 形式のイラストは新規に使用しない。
- ヒーローや各セクションに配置する装飾イラストは、用途に応じて個別PNGまたは複数のポイントイラストをまとめたPNGシートとして管理できる。UIテキストは画像内に埋め込まない。
- 写真バナーなど複数情報を含む主要ビジュアルは、用途が明確なPNGバナーとして `public/banners/` 配下で管理できる。ただしデザイン上必要な場合に限り使用し、無理に全体バナーとして埋め込まない。
- FVおよび各セクションでは、生成PNG `public/illustrations/kawabit-spot-*.png` をポイントイラスト素材として使用し、表示時に画像のアスペクト比を変更しない。
- サービスカードや理由セクションなどの小アイコン・イラストは、CSS疑似要素やSVGではなく、生成PNGを `public/icons/` または `public/illustrations/` 配下で管理して表示する。
- ナビゲーション、見出し、本文、ボタン、電話番号、URLなどのUIテキストは画像内に埋め込まず、HTML/CSSで実装する。
- 生成PNGは `public/illustrations/` 配下に配置し、用途が分かるファイル名にする。

## 開発コマンド

- `npm run dev`: 開発サーバーを起動する。
- `npm run build`: 本番ビルドを実行する。
- `npm run start`: ビルド済みアプリケーションを起動する。
- `npm run lint`: Next.js の lint を実行する。

## ステアリングファイル構成

今後このプロジェクトでは、要件・設計・実装タスクを必要に応じて以下のファイルに分割して管理する。

1. `requirements_definition.md`
   - プロダクト要件定義書
   - プロダクトビジョンと目的
   - ターゲットユーザーと課題・ニーズ
   - 主要な機能一覧
   - 成功の定義
   - ユーザーストーリー
   - 受け入れ条件
   - 機能要件
   - 非機能要件

2. `requirements.md`
   - 今回の作業単位での要求内容
   - 変更・追加する機能の説明
   - ユーザーストーリー
   - 受け入れ条件
   - 制約事項

3. `design.md`
   - 今回の変更内容の設計
   - 実装アプローチ
   - 変更するコンポーネント
   - データ構造の変更
   - 影響範囲の分析

4. `tasklist.md`
   - 具体的な実装タスク
   - タスクの進捗状況
   - 完了条件

## AGENTS.md 運用ルール

- `AGENTS.md` は、プロジェクトの現状を表す北極星ドキュメントとして扱う。
- 機能追加、仕様変更、運用方針変更が発生した場合、Codex は必要に応じてこのファイルを更新し、記述を最新状態に保つ。
- 要件整理、仕様確認、実装判断が必要な作業では、`requirements_definition.md` を参照する。
- `AGENTS.md` と `requirements_definition.md` に差分が生じた場合は、両方を同時に更新して整合を保つ。
- 作業単位で詳細な変更要件が必要な場合は `requirements.md` を作成または更新する。
- 実装前に設計の明文化が必要な場合は `design.md` を作成または更新する。
- 実装を分解して進捗を管理する場合は `tasklist.md` を作成または更新する。

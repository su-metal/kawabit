# 開発ガイドライン

このドキュメントは、プロジェクトの現状と運用方針を示す北極星ドキュメントとして扱う。

## プロジェクト概要

- このプロジェクトは Next.js 14 と React 18 を使用する Web アプリケーションである。
- 現在のアプリケーションは、川辺コンピューターとかわべ家庭教師学院の2つの専門サービスへ利用者を案内する「KAWABIT（カワビット）」ブランドのハブサイトである。
- トップページでは、パソコン修理・データ復旧の相談先として `https://kawabecom.jp/`、プロ家庭教師・学習相談の相談先として `https://kawabegakuin.jp/` へ誘導する。
- サイトデザインは `design.md` を正本とし、白〜生成りの余白、黒の極太ライン、2〜4色程度のソリッドなフラットカラー、ミニマルなコミック風PNGイラストを基調とする。
- トップページのFVは、中央にメインコピー「学びもITも、ここから相談。」、その下に大きなブランドイラスト `public/illustrations/used/kawabit-hero-wide-flow-v1.png`、画面下部に家庭教師・パソコン修理の2つのサービスカードを配置し、初期表示内で両サービスの並列認知と専門サイトへの入口を成立させる。
- トップページの続きは、すべてをイラストで説明せず、見出し・本文・CTAを主役にしながら、各セクションへ極太ラインのPNGイラストをポイントとして配置する。
- FVの直後には「川辺ブランドの歩み」セクションを置き、1997年のかわべ家庭教師学院、1999年の川辺コンピューター、現在の2つの専門サービスを控えめな年表カードで表示する。KAWABIT 自体の役割説明ではなく、両サービスの存在を自然に記憶へ残すための沿革紹介として扱う。
- FV内の2つのサービスカードは、サービス詳細セクションへ遷移し、対応する家庭教師またはパソコン修理のタブを自動で表示する。
- FVはヘッダーを除いた初期表示領域を基本にしつつ、モバイルでは縦積みで読みやすさを優先する。
- ヘッダーはスクロール状態に応じて変化し、スクロール前（一番上の状態）では「KAWABIT — 学びもITも、ここから相談。」というテキストが横スクロールし続ける黄色の「流れる帯（ティッカー）」になり、少しでもスクロールを開始すると、元のロゴ・メニュー（主要ナビゲーションおよび相談ボタン）がある通常のヘッダーに切り替わる。
- 表示言語は日本語を基本とし、問い合わせ時間は `10:00-19:00 / 日曜休` として表示する。
- アプリケーションコードは主に `app/` に配置する。
- 静的アセットは `public/` に配置する。
- 画像や素材の元データは、必要に応じて `asset-sources/` に配置する。

## サイト方向性

KAWABIT のトップページは「入口（ハブ）」として機能させることを最上位の目的とし、短期のコンバージョン最大化を目的としたLPとしては扱わない。改修・機能追加・コピー調整を判断する際は、以下のスタンスを基準にする。

- **両サービスの並列認知を最優先する**: 訪問者がどちらか片方の目的（例: 家庭教師の検索、PC修理の検索）で来訪した場合でも、もう一方のサービスの存在を必ず認知できる状態にする。来訪目的のサービスだけを強調し、反対側のサービスを目立たなくする構成・コピー・装飾は採用しない。
- **「頭の片隅に残す」を狙う**: 反対側のサービスは、その場で即決させるための強い訴求ではなく、「KAWABIT には学びと IT の両方がある」と軽く印象付ける程度で十分とする。情報量・主張の強さは、来訪目的側と均衡させる。
- **長期記憶に残すことを成功と見なす**: 直近の問い合わせ件数や即時クリックの最大化ではなく、「数年後に必要が生じたときに KAWABIT を思い出してもらう」ことを成功の中心に置く。短期 KPI を理由に押しの強い演出やしつこい再訴求を追加しない。
- **押し売り・煽りを避ける**: 限定オファー、カウントダウン、誇張表現、強い不安喚起、勧誘色の強い反復CTA など、短期コンバージョン最適化型の手法は採用しない。落ち着いたトーンを保ち、「相談したくなったときに選びやすい存在」であることを優先する。
- **入口に徹し、詳細は専門サイトに委ねる**: 料金、施工事例、講師プロフィール、実績数値などの深い情報は `https://kawabecom.jp/` と `https://kawabegakuin.jp/` の責任範囲とする。トップページでは「どちらに相談すべきかの判断」と「専門サイトへの移動」だけを支援し、専門サイトの情報を不要に複製しない。
- **改修時の判断チェック**: 新しい要素を追加・変更する際は、(1) 両サービスの並列認知を弱めないか、(2) 数年後の想起に寄与するか/逆行しないか、(3) 短期コンバージョン目的の押し付けになっていないか、を満たすかで判断する。満たさない場合は採用を見送るか、入口としての性格を保つ形に再設計する。

### 想定する流入経路とクロス認知の狙い

KAWABIT トップページの主要な役割は、「片方のサービスを目的に流入したユーザーに、もう一方のサービスの存在を軽く認知させ、将来の潜在顧客として残す」ことにある。具体的な想定経路は以下。

- **`https://kawabegakuin.jp/`（かわべ家庭教師学院）側から流入したユーザー**: 学習相談や家庭教師を探している。KAWABIT を経由することで、「同じ運営が PC 修理・データ復旧もやっている」ことを頭の片隅に残してもらう。今すぐ PC 修理を必要としていなくても、数年後にパソコンが故障した・買い替えたいなどの状況になったときに「そういえば KAWABIT で見た覚えがある」と想起してもらえる状態を狙う。
- **`https://kawabecom.jp/`（川辺コンピューター）側から流入したユーザー**: PC 修理・データ復旧・PC 購入相談を探している。KAWABIT を経由することで、「同じ運営が家庭教師サービスもやっている」ことを頭の片隅に残してもらう。子どもの進学・受験・学習面の悩みが将来発生したときに、家庭教師の選択肢として KAWABIT を想起してもらえる状態を狙う。

このため、トップページでは「来訪目的のサービス」だけを強調せず、反対側のサービスも同等の存在感で並列に提示する。即時の遷移を強要せず、「もう一方もある」という事実が落ち着いて頭に残るレイアウト・コピーを優先する。

## ビジュアルアセット運用ルール

- サイト内のイラストは PNG 形式のラスター画像として生成・配置する。
- SVG 形式のイラストは新規に使用しない。
- ヒーローや各セクションに配置する装飾イラストは、用途に応じて個別PNGまたは複数のポイントイラストをまとめたPNGシートとして管理できる。UIテキストは画像内に埋め込まない。
- 写真バナーなど複数情報を含む主要ビジュアルは、用途が明確なPNGバナーとして `public/banners/` 配下で管理できる。ただしデザイン上必要な場合に限り使用し、無理に全体バナーとして埋め込まない。
- FVでは `public/illustrations/used/kawabit-hero-wide-flow-v1.png` を大きなブランドイラストとして使用し、同じ初期表示内に生成PNGサムネイル付きの2つのサービスカードを配置する。各セクションでは生成PNG `public/illustrations/kawabit-spot-*.png` をポイントイラスト素材として使用する。表示時はいずれも画像のアスペクト比を変更しない。
- サービスカードや理由セクションなどの小アイコン・イラストは、CSS疑似要素やSVGではなく、生成PNGを `public/icons/` または `public/illustrations/` 配下で管理して表示する。
- ナビゲーション、見出し、本文、ボタン、電話番号、URLなどのUIテキストは画像内に埋め込まず、HTML/CSSで実装する。
- 生成PNGは `public/illustrations/` 配下に配置し、用途が分かるファイル名にする。

## 開発コマンド

- `npm run dev`: 開発サーバーを起動する。
- `npm run build`: 本番ビルドを実行する。
- `npm run start`: ビルド済みアプリケーションを起動する。
- `npm run lint`: Next.js の lint を実行する。

## ステアリングファイル構成

今後このプロジェクトでは、要件・設計・実装タスクを必要に応じて以下のファイルに分割して管理する。

1. `requirements_definition.md`
   - プロダクト要件定義書
   - プロダクトビジョンと目的
   - ターゲットユーザーと課題・ニーズ
   - 主要な機能一覧
   - 成功の定義
   - ユーザーストーリー
   - 受け入れ条件
   - 機能要件
   - 非機能要件

2. `requirements.md`
   - 今回の作業単位での要求内容
   - 変更・追加する機能の説明
   - ユーザーストーリー
   - 受け入れ条件
   - 制約事項

3. `design.md`
   - 今回の変更内容の設計
   - 実装アプローチ
   - 変更するコンポーネント
   - データ構造の変更
   - 影響範囲の分析

4. `tasklist.md`
   - 具体的な実装タスク
   - タスクの進捗状況
   - 完了条件

## AGENTS.md 運用ルール

- `AGENTS.md` は、プロジェクトの現状を表す北極星ドキュメントとして扱う。
- 機能追加、仕様変更、運用方針変更が発生した場合、Codex は必要に応じてこのファイルを更新し、記述を最新状態に保つ。
- 要件整理、仕様確認、実装判断が必要な作業では、`requirements_definition.md` を参照する。
- `AGENTS.md` と `requirements_definition.md` に差分が生じた場合は、両方を同時に更新して整合を保つ。
- 作業単位で詳細な変更要件が必要な場合は `requirements.md` を作成または更新する。
- 実装前に設計の明文化が必要な場合は `design.md` を作成または更新する。
- 実装を分解して進捗を管理する場合は `tasklist.md` を作成または更新する。
